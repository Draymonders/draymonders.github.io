# ClickHouse 使用指南

ClickHouse 是一个用于联机分析 (OLAP) 的列式数据库管理系统 (DBMS)。作为资深开发者，理解 ClickHouse 不仅仅是会写 SQL，更重要的是理解其底层的**列式存储**、**向量化执行**以及**MergeTree**引擎机制。

## 1. 导读

ClickHouse 查询快的本质是软件工程对硬件性能的压榨做到了极致。

### 1.1 核心特性

- **列式存储 (Column-Oriented)**：数据按列存储，分析查询时只读取必要的列，极大减少 IO。同时列存带来了极高的压缩率（通常 10:1 以上）。
- **向量化执行 (Vectorized Execution)**：利用 CPU 的 SIMD (Single Instruction, Multiple Data) 指令集，一次处理一批数据，而非一行行处理。
- **MPP 架构 (Massively Parallel Processing)**：查询在所有可用分片上并行执行，利用多核多机能力。
- **无事务 (No Transactions)**：为了极致的写入和查询性能，牺牲了完整的 ACID 事务支持（虽支持部分原子性，但非传统 OLTP 意义上的事务）。

### 1.2 设计哲学

ClickHouse 是典型的 **Bottom-up** 设计。它没有试图用一套通用的算法解决所有问题，而是针对特定的数据类型、特定的查询场景，选择了最适合的算法（例如针对不同基数的 Group By 有不同的哈希表实现）。

---

## 2. 使用场景

ClickHouse 不是万能的，适合做数据分析，并且查询QPS不能太高。

### 2.1 最佳实践

*   **日志与事件流分析**：如用户点击流、Nginx/App 日志。特点是：数据量极大、追加写入、极少更新。
*   **实时监控指标 (Metrics)**：Prometheus 的后端存储方案之一。
*   **BI 报表与 Ad-hoc 查询**：并在海量宽表上进行秒级聚合响应。
*   **用户画像与标签筛选**：利用 BitMap 等函数进行快速的人群圈选。

### 2.2 反模式

*   **OLTP 事务**：不要用它做订单库、用户余额库。ClickHouse 不支持高频的单行 Update/Delete。
*   **Key-Value 检索**：虽然支持主键查找，但高并发的点查（Point Query）不是其强项，Redis/HBase 更合适。
*   **复杂的关联查询**：虽然支持 Join，但大表 Join 大表性能不可控，推荐在 ETL 阶段处理成大宽表 (Denormalization)。

---

## 3. 存储结构

理解存储结构是调优 ClickHouse 的关键。这里以最核心的 **MergeTree** 引擎家族为例。

### 3.1 目录与文件结构

在磁盘上，MergeTree 表的数据是按 **分区 (Partition)** 组织的。每次批量写入（Insert）都会生成一个新的临时数据片段 (**Data Part**)，后台任务会不断将小片段合并（Merge）成大片段。

典型的目录结构：
```text
/var/lib/clickhouse/data/db_name/table_name/
├── 20230101_1_1_0/          <-- Data Part (分区名_最小块_最大块_层级)
│   ├── checksums.txt
│   ├── columns.txt
│   ├── count.txt
│   ├── primary.idx          <-- 主键索引 (稀疏索引)
│   ├── [Column].bin         <-- 数据文件 (压缩后的列数据)
│   └── [Column].mrk2        <-- 标记文件 (连接索引与数据的桥梁)
└── ...
```

### 3.2 核心组件

1.  **`.bin` 文件**：存储实际的列数据。数据在 `.bin` 文件中是被压缩和排序的。
2.  **`.mrk` (Mark) 文件**：非常关键。由于 `.bin` 是压缩的，无法直接定位。Mark 文件记录了数据块（Block）在 `.bin` 文件中的物理偏移量。它像是一个“索引的索引”。
3.  **LSM-Tree 思想**：ClickHouse 的写入机制类似于 LSM-Tree，数据先顺序写入，不可变，依靠后台 Merge 线程进行压实（Compaction），清理标记为删除的数据（TTL 或 CollapsingMergeTree）。

## 4. 索引机制

ClickHouse 的索引设计非常独特，它不使用传统的 B+ 树，而是使用 **稀疏索引 (Sparse Index)**。

### 4.1 主键索引

*   **稀疏性**：默认每 `8192` 行（`index_granularity`）才生成一个索引条目。
*   **原理**：索引存储的是每个 Granule（颗粒）的第一行数据。
*   **优势**：索引非常小，常驻内存。亿级数据的主键索引可能只有几十 MB。
*   **劣势**：无法精确地位到某一行，只能定位到包含该行的 Granule（通常是 8192 行的一个范围），然后扫描解压这个范围的数据。这就是为什么它不适合单行点查。

**查询流程**：

1.  通过 `primary.idx` 找到目标数据可能存在的 Granule 范围。
2.  通过 `.mrk` 文件找到该 Granule 在 `.bin` 文件中的压缩块偏移量。
3.  读取 `.bin` 文件，解压相关块，并在内存中过滤出需要的数据。

### 4.2 跳数索引

除了主键，ClickHouse 支持二级索引，称为跳数索引。它的目的不是直接定位，而是**排除**。

*   **minmax**：记录一段范围内列的最大最小值。如果查询条件不在范围内，直接跳过。
*   **set**：记录一段范围内列的所有去重值（适合基数小的列）。
*   **bloom_filter**：利用布隆过滤器快速判断值是否**不**存在。

> **资深建议**：不要迷信二级索引。在 ClickHouse 中，精心设计的 `ORDER BY`（主键）通常比任何二级索引都有效。只有在过滤效果极强（能跳过 90% 以上数据）时，才考虑 Skipping Index，否则其构建和维护成本可能大于收益。

## 5. 物化视图

ClickHouse 的物化视图是一种非常强大的**预计算**机制，它本质上是一个**插入触发器 (Insert Trigger)**。

### 5.1 工作原理

当数据被写入源表（Source Table）时，物化视图会拦截这批写入数据，按照预定义的 SQL 逻辑进行计算（如 `GROUP BY`, `WHERE` 过滤），然后将结果写入到一张隐含的目标表（Target Table）中。

> **注意**：物化视图只对**新写入**的数据生效，不会自动追溯历史数据（除非手动执行 `INSERT INTO ... SELECT`）。

### 5.2 核心价值：空间换时间

*   **预聚合**：对于极其消耗资源的聚合查询（如“计算每天的 UV”），物化视图可以在数据写入时就完成计算。查询时直接读结果表，速度提升可达千倍。
*   **数据流转**：将数据从一种引擎同步到另一种引擎（如从 Log 引擎同步到 Kafka 引擎）。

### 5.3 物化视图 vs 索引


| 特性 | 索引 (Index) | 物化视图 (Materialized View) |
| :--- | :--- | :--- |
| **本质** | 原始数据的**目录** | 原始数据的**副本或衍生品** |
| **数据形态** | 不改变数据，只提供快速定位路径 | **改变数据**（可以是聚合后的、过滤后的、重塑结构的） |
| **存储消耗** | 较小（通常是稀疏的） | 较大（取决于聚合程度，是一张真实的物理表） |
| **查询方式** | `SELECT * FROM table WHERE ...` (隐式使用) | `SELECT * FROM mv_table` (显式查询视图表) |
| **加速原理** | **减少扫描范围** (Skip Data) | **减少计算量** (Pre-compute) |

> **资深建议**：如果你的查询慢是因为**扫描的数据量太大**，请优化索引（主键）；如果是因为**计算（聚合/排序）太重**，请使用物化视图。
